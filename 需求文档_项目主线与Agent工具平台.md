## 1. 产品愿景

### 1.1 要解决什么问题

用户在真实业务中同时需要：**领域文档（规范、报告、论文）**、**可重复的数据实验（多方案、多模型）**、以及**可组合的智能动作（问答、推理、批处理）**。本平台将三者统一在**同一项目空间**内：先沉淀知识与数据实验，再通过 **Agent** 把「预置算法 + 用户自训模型 + 检索」编排成**用户自己的自动化工作流**。

### 1.2 核心价值

- **一条主线**：从「立项」到「知识 → 子课题实验 → 模型资产 → Agent 调用」全链路可追溯。
- **预置 + 自训**：系统提供可选的深度学习 / 机器学习**模型模板与推理入口**；用户训练产物可**注册为可调工具**。
- **结构清晰**：**一个项目对应一个知识库**；项目下可建多个**子项目**（实验单元），每个子项目独立管理数据实验、多张分析图、多个已保存模型。
- **预测与 Agent 一体**：保存后的模型既支持**传统界面手动上传预测集**批量预测，也支持在 **Agent 中通过 Tool** 上传预测集、驱动数据处理与训练/推理编排；**用户切换项目/子项目**即切换知识库范围与可用模型工具。
- **可观测的 Agent**：交互过程**可视化**（时间线、工具调用、附件状态、终端式输出），而非纯文本黑盒。

### 1.3 非目标（边界）

- 不承诺开箱即用的行业垂直 SOTA，而是提供**可扩展模型目录**与**统一注册协议**。
- 多租户强隔离、金融级审计可在架构上预留，首版以**单租户/单团队部署**为主。

## 2. 领域模型（核心）

### 2.1 概念层级

```
项目 (Project) — 顶层工作空间
├── 知识库 (Knowledge Base) — 与项目 1:1，存储 PDF/TXT 等，支持检索（RAG）
├── 元数据：名称、描述、标签、创建人、时间
│
├── 子项目 (SubProject / Experiment Unit) — 可多个
│   ├── 数据资产：原始表、清洗版本、特征/目标配置快照
│   ├── 训练与模型：多次训练运行；多个「已保存模型」版本（权重/参数 + 预处理管道）
│   ├── 可视化与附件：分析图、相关性结果图、说明文字（图片 + 元数据）
│   └── 可选：子项目级备注、实验假设标签
│
└── Agent 工作流上下文：由用户显式选择 **当前项目 + 可选子项目** → 决定知识库范围与注入的模型 Tool 列表
```

**约束与说明**：

| 关系 | 规则 |
|------|------|
| 项目 : 知识库 | **严格 1:1**；删除项目需定义知识库是否级联删除或归档 |
| 项目 : 子项目 | **1 : N**；子项目之间相互隔离数据路径，避免误覆盖 |
| 子项目 : 已保存模型 | **1 : N**；同一子项目内可对比多个模型版本 |
| 子项目 : 图片/分析资产 | **1 : N**；每条资产带类型（如 correlation / eda / training_curve）、可选关联某次训练 run_id |

### 2.2 标识与追溯

| 实体 | 建议标识 | 追溯要求 |
|------|-----------|-----------|
| 项目 | `project_id` | — |
| 知识库 | `kb_id`（可与 `project_id` 相同或一对一映射表） | 文档变更版本、索引重建时间 |
| 子项目 | `sub_project_id` | 必须归属某 `project_id` |
| 训练运行 | `run_id` | 归属 `sub_project_id`；记录数据版本、算法模板 id、超参、随机种子 |
| 已保存模型 | `model_id` | 归属 `sub_project_id`；记录 `source_run_id`、指标、导出时间 |
| 分析图/附件 | `artifact_id` | 归属 `sub_project_id`；可选 `run_id` |

## 3. 功能需求

### 3.1 预置模型目录（系统提前准备）

**目标**：用户无需从零实现算法，即可在训练与推理时选择**系统维护的模型族**（经典 ML + 深度学习等）。

**要求**：

- **模型注册表**（元数据）：`template_id`、名称、类型（回归/分类/…）、输入约定（表格特征 / 图像 / 序列等）、依赖（PyTorch/sklearn/…）、资源标签（CPU/GPU）、是否支持多输出等。
- **选择与配置**：创建训练任务时从目录中选取模板，填写超参数（JSON Schema 校验）；部分模板仅支持推理（预训练权重由平台挂载）。
- **版本管理**：`template_id + semver`；重大变更保留旧版一段时间以保证旧实验可复现。
- **扩展**：支持后续通过插件或独立服务注册新模板（与 MCP / HTTP 推理服务对接时在实现层完成，规格上保留扩展点）。

### 3.2 项目与知识库（1:1）

**项目**：

- 创建 / 重命名 / 描述 / 标签 / 归档或删除。
- 首页或仪表盘展示：下属子项目列表、最近模型、知识库文档数量等。

**知识库（绑定到该项目）**：

- 上传 **PDF、TXT**（可扩展 Markdown、DOCX 等）。
- 解析、分块、建立索引（首版可为 TF-IDF，演进为向量检索）。
- 检索时可限定「仅当前项目知识库」，供 **RAG** 与 **Agent** 使用。
- **禁止**同一项目挂载多个并列「主知识库」；若需资料分区，可用「集合/文件夹」在逻辑层划分，仍属同一 `kb_id`。

### 3.3 子项目：训练、多模型与可视化资产

**子项目**（名称建议：**实验单元 / 子课题**，避免与顶层「项目」混淆）：

1. **创建与归属**：必须指定父 `project_id`；可填写子项目名称、目标说明。
2. **数据**：在子项目内上传或引用数据文件（CSV/Excel 等）；支持数据版本与基本质量校验（与旧版数据规范可复用）。
3. **训练**：
   - 选择**预置模型模板** + 特征/目标 + 划分比例 + 超参；
   - 异步训练；每次执行产生 `run_id`，持久化指标与日志。
4. **保存模型**：从某次成功的 `run` 将权重与预处理参数**注册为「子项目下的模型实例」**，用户命名（如 `工艺_v2`）；**同一子项目可保存多个模型**。
5. **批量预测（手动路径，与旧版平台一致）**：
   - 用户在**子项目工作台 / 我的模型**中选中某已保存模型；
   - **上传预测集文件**（CSV/Excel），文件仅需包含与训练时一致的**输入特征列**（无目标列亦可）；
   - 服务端加载预处理管道与模型，输出带预测列的结果文件供**下载**；
   - 需校验列名、类型与缺失策略，错误信息明确可定位。
6. **图片与分析资产**：
   - 上传或系统自动生成（如相关性热力图、散点图、损失曲线）；
   - 元数据：标题、类型、关联 `run_id`（可选）、说明文字；
   - 支持列表、预览、下载、删除。

### 3.3.1 Agent 内预测集与「模型即工具」的增强（v1.1）

在 **3.3 第 5 点** 的手动批量预测之外，平台须将能力 **封装为 Tool**，使 Agent 可在对话流中完成同类任务，并扩展至数据处理与训练编排：

| 能力 | 说明 |
|------|------|
| **预测集上传（会话附件）** | 用户在 Agent 界面**上传预测集文件**（与手动预测相同格式约束）；文件绑定当前会话（及上下文中的 `project_id` / `sub_project_id`），供后续 Tool 引用（如 `file_ref` 或临时存储 id）。 |
| **批量推理 Tool** | 对指定 `model_id` + 已上传预测集执行批量预测，返回**结果文件路径/下载链接**与行数摘要；Tool 的 JSON Schema 中明确 `model_id` 可选枚举为**当前子项目下已注册模型**（由上下文注入，避免模型幻觉填错 id）。 |
| **数据处理 Tool** | 对会话内上传的表格做预览、列校验、简单清洗（与平台数据规范一致）；可被 Agent 在推理前调用。 |
| **训练编排 Tool**（可选/策略控制） | 在已绑定子项目上下文中，触发一次训练 run（预置模板 + 超参）；属高风险操作，需**确认策略**或仅对高级角色开放。 |

**灵活性要求**：

- Agent 应能通过 **多步 function calling** 组合：例如「校验预测集列 → 调用用户模型批量推理 → 将结果摘要写入回复」或「读取训练数据 → 启动训练 → 再对预测集推理」。
- **上下文强制**：当用户切换 **项目** 时，RAG 检索范围切换为该项目知识库；切换 **子项目** 时，动态 Tool 列表仅包含**该子项目下的已保存模型**及该项目级系统工具（见 3.4.1）。禁止在未绑定项目时访问任意项目数据。

### 3.4 Agent：工具化模型与用户工作流

**目标**：Agent 不仅是聊天，而是能**调用平台能力**完成「检索 → 推理 → 汇总」等步骤；其中**用户训练并注册的模型**与**预置推理接口**统一为 **Tool / Function Calling** 形态。

**工具类型（逻辑分类）**：

| 类别 | 示例 | 说明 |
|------|------|------|
| 系统内置 | `search_project_kb`、`list_subprojects`、`get_dataset_preview` | 只读、低风险 |
| 预置模型推理 | `infer_template_{template_id}` | 由平台实现，参数校验后执行 |
| **用户模型推理** | `infer_user_model` / `batch_predict_with_file` | **训练产物注册后自动生成**；入参含 `model_id` 与**会话内预测文件引用**；强制校验与当前上下文 `project_id`/`sub_project_id` 一致 |
| 训练类（可选） | `start_training_run` | 高权限，需确认或策略限制 |
| 代码/沙箱（可选演进） | `execute_python_sandbox` | 受资源与路径约束 |

**与「上一版平台」批量预测的关系**：子项目工作台中的**手动上传预测集**与 Agent 内的 **batch 推理 Tool** 须**复用同一套后端推理、列校验与预处理逻辑**，仅入口不同（表单 vs 会话附件 + function calling）。

**与「模型即工具」的约束**：

- 每个 `model_id` 暴露的 Tool 定义包含：**JSON Schema 输入**（特征名与类型）、**输出说明**、**延迟与资源提示**；批量推理类 Tool 须支持**会话文件引用**入参。
- 删除子项目或模型时，同步吊销对应 Tool 注册或标记为不可用。

#### 3.4.1 会话上下文：项目 + 子项目绑定

| 会话字段 | 作用 |
|-----------|------|
| `project_id`（必选） | 决定 **知识库 RAG** 检索范围（该项目 1:1 知识库）。 |
| `sub_project_id`（可选但推荐） | 决定 **可用用户模型 Tool 列表**、默认数据路径、会话附件上传目录；未选时仅展示项目级工具（如知识检索、子项目列表），**不暴露**具体子项目模型推理 Tool，避免误调用。 |

**交互**：Agent 页面提供**项目选择器**与**子项目选择器**（级联或下拉）；变更时**清空或重新确认**会话附件与已排队工具调用（可选产品策略）；**后端**在每次 `turn` 请求中校验会话上下文与 Token 中声明的 id 一致。

**工作流**：

- 用户在绑定 **项目 + 子项目** 的会话中，通过自然语言触发多步工具调用；模型可组合：**知识检索 → 数据预览/校验 → 批量预测（含上传文件）→ 训练（若策略允许）**。
- **默认知识范围**：当前 `project_id` 对应知识库；子项目不改变知识库归属，但**收窄**模型与数据 Tool 的作用域。
- **Memory**：会话级 + 项目级摘要（用户确认后写入），避免幻觉污染（见安全策略）。

#### 3.4.2 Agent 交互过程可视化（必选）

Agent 界面**不得**仅展示最终文本回复，须同时呈现**可观测的执行过程**，满足调试、审计与心理安全感：

| 可视化元素 | 内容要求 |
|-------------|-----------|
| **对话时间线** | 用户消息与助手消息分区展示；助手侧可折叠。 |
| **工具调用卡片** | 每一步 function calling 展示：**工具名称**、**入参摘要**（可展开 JSON）、**执行状态**（进行中/成功/失败）、**耗时**；失败时展示错误摘要与可重试提示（若适用）。 |
| **附件与文件** | 用户上传的预测集/数据文件在侧栏或时间线中显示**文件名、大小、解析状态**；工具生成结果文件时提供**下载入口**或内联预览（如表格前 N 行）。 |
| **终端/日志区** | 类终端面板展示工具 stdout、系统日志片段（脱敏）；支持清空、暂停自动滚动、一键复制。 |
| **流式输出** | 最终自然语言回答支持流式打字效果；与工具卡片时序一致（工具完成后输出正文）。 |
| **上下文指示条** | 顶部或侧栏常驻显示：**当前项目名 / 子项目名**，便于用户确认「当前 Agent 作用于哪套知识库与哪套模型」。 |

事件协议建议与后端 **SSE/WebSocket 统一 JSON 事件类型**（如 `tool_start`、`tool_result`、`file_ready`、`terminal_stdout`、`message_delta`、`done`），前后端契约写进 OpenAPI 或附属 schema。

#### 3.4.3 Agent 界面：会话管理与上下文管理（必选）

Agent 页不仅是「单会话聊天」，须具备与专业控制台一致的**会话生命周期**与**作用域显式管理**，避免用户在不自知的情况下跨项目误操作。

**会话管理（UI + 行为）**：

| 能力 | 要求 |
|------|------|
| **会话列表** | 侧栏或独立面板列出当前用户在该环境下的会话；每条显示**标题/摘要**（首条用户消息或用户可编辑标题）、**最近更新时间**、可选**固定/置顶**。 |
| **新建会话** | 显式「新建」入口；新建时可继承当前 **项目/子项目** 选择，或由用户先选上下文再开聊（产品二选一，须在界面上一致）。 |
| **切换会话** | 点击切换当前活动会话；**加载该会话历史消息**与**该会话已绑定的附件列表**；UI 须标明当前会话 id 或名称，避免与浏览器多标签混淆。 |
| **重命名 / 删除** | 支持重命名会话；支持删除（或归档）会话，**二次确认**；删除后会话内附件按策略清理或仅解除 UI 引用（实现层定义，须在帮助文案中说明）。 |
| **空状态** | 无会话时引导创建会话或跳转项目；有会话但列表为空分支不成立时的占位文案保持工程化风格（见 §5）。 |

**上下文管理（与 3.4.1 配合）**：

| 能力 | 要求 |
|------|------|
| **项目 / 子项目选择器** | **常驻可见**（侧栏顶部或对话区上方 **上下文条**）；变更时按 3.4.1 规则**收窄 RAG 与 Tool**；若变更会导致附件或排队工具失效，须**弹窗或条内提示**并让用户确认。 |
| **上下文只读摘要** | 在对话区附近展示**当前项目名、子项目名**（子项目未选时明确标注「未绑定子项目 — 无用户模型 Tool」），与 3.4.2 **上下文指示条**一致。 |
| **附件与上下文** | 会话附件列表须显示**归属会话**；切换会话后附件区仅显示**当前会话**文件，避免混用。 |
| **可选增强** | 「清空本轮对话但保留上下文」「复制会话 id」等可作为后续迭代，首版至少保证**选择器 + 指示 + 会话切换**三者完备。 |

## 4. 非功能需求

### 4.1 性能与可扩展

- 训练任务异步队列；推理 P95 延迟目标按部署环境文档化。
- 模型目录与 Tool 列表支持分页/按项目过滤，避免单次请求撑满上下文。

### 4.2 安全

- 知识库与数据路径隔离；工具调用**禁止跨项目任意路径**。
- 用户模型 Tool 默认仅同一项目内可调用；跨项目调用需显式授权（可选功能）。
- 敏感配置（API Key）环境变量管理；审计日志记录工具名、参数摘要、结果状态。

### 4.3 可维护性

- 元数据与工件存储分离：小 JSON/DB 存索引，大文件存对象存储或本地分目录。
- OpenAPI / 事件 schema 版本化，便于前后端联调。

## 5. 用户界面信息架构与视觉规范

### 5.0 页面结构与导航

- **顶层**：项目列表 → 进入项目总览。
- **项目内 Tab**：
  - 知识库（文档与检索测试）
  - 子项目列表 → 进入子项目工作台
- **子项目内 Tab**：
  - 数据
  - 训练与运行历史
  - 已保存模型
  - 分析图与附件
- **Agent**（布局建议）：
  - 左侧：**会话管理区**（列表、新建、切换、重命名、删除）+ **项目/子项目选择器** + **上下文指示条**（见 **3.4.3**）；
  - 中部：**对话时间线**（含工具卡片与附件状态）；
  - 右侧或底部：**终端/日志** + **本地上传区**（预测集、数据文件）；附件仅作用于**当前会话**；
  - 支持响应式：窄屏下终端区可折叠为抽屉；会话列表可收折为图标条 + 抽屉。

### 5.1 视觉风格定位：工业控制台 / 工控台（Industrial Console）

**角色设定**：界面应让用户感到正在操作**可靠、可长期运行的系统设备**——耐用、可维护、专业硬核；参考制造/物联网/能源/仓储控制台、DevOps 监控仪表板、强功能感 B2B 工具，而非消费级圆润插画风。

**整体氛围**：坚固、工程化、可信赖；信息密度**可偏高**但排布必须清晰（分区、对齐线、层级一眼可辨）；**警示色仅用于关键状态与告警**，避免满屏花哨。

**背景与材质（建议实现）**：

- 主背景：**深色基底**（炭黑 `#0d0f12`～`#12151a`、铁灰层次），可叠加**细网格**、微弱水平条纹或**轻噪点**模拟金属/喷涂面板，忌大面积纯平死黑。
- 分区：使用**低透明度描边**（如 `rgba(255,255,255,0.06~0.12)`）界定面板与卡片；可选极淡的**斜纹警示带（hazard stripes）**仅用于「危险操作 / 训练 / 删除」区域条头，**禁止滥用**。
- 可选装饰：顶栏或侧栏下缘的**刻度线**、**导轨/螺栓**式 1px 点缀（极简），增强「机柜/设备感」而不抢信息。

---

### 5.2 色彩系统

| 语义 | 建议方向 | 用途 |
|------|-----------|------|
| 基底 | 炭黑 / 铁灰渐变或分层 | 页面背景、侧栏、主面板底 |
| 表面 | 略亮于基底的 `#161a20`～`#1e2329` | 卡片、表格、模态框 |
| 主高光（冷色） | **青 / 冰蓝**（如 `#22d3ee`、`#38bdf8`） | 主按钮、焦点环、当前 Tab、关键指标强调 |
| 次要高光 | 冷灰蓝描边 | 默认边框、hover 描边增强 |
| **警示 / 告警** | **琥珀 / 橙**（如 `#f59e0b`、`#fb923c`） | 训练失败、错误、需确认操作；**告警闪烁须低频、可关闭** |
| 成功 | 偏冷的绿或青绿（如 `#34d399`） | 成功状态、完成 |
| 文本 | 主文 `#e8eaed`，次级 `#9ca3af`，禁用 `#6b7280` | 正文、说明、占位 |

**原则**：冷色高光承担「可操作、当前焦点」；琥珀/橙承担「注意与风险」；**避免**彩虹渐变与强发光阴影堆叠。

---

### 5.3 排版与字体

- **字体**：无衬线、偏紧凑；标题与导航可用 **半粗体 / 粗体**；数据区、标签、表头可使用 **全大写或小型大写（letter-spacing）** 增强「仪表/铭牌」感（如 `STATUS`、`RUN ID`），但**正文说明保持正常大小写**以保证可读性。
- **字号层级**：至少 3 级（页面标题 / 区块标题 / 正文与表格）；行高略紧于消费级 UI，以承载表格与日志。
- **网格与对齐**：主要内容区采用 **8px 或 4px 基准网格**；表格与工具卡片左对齐；关键数字**右对齐**（指标、耗时）。

---

### 5.4 组件与面板（质感）

- **卡片 / 面板**：**硬边或小圆角**（如 2～6px）；边框清晰，**短而锐利**的阴影（如 `0 1px 0 rgba(255,255,255,0.04) inset` + 极浅外阴影），避免「飘浮卡片」式软阴影。
- **表格**：斑马纹可选（极低对比）；表头与数据行分隔线明确；选中行用**左侧冷色条**或**背景微亮**表示。
- **按钮**：主按钮为实心冷色 + 细边框；次要按钮为描边透明底。**Hover**：描边与边框略增亮、阴影略加深。**Active**：轻微下沉（`translateY(1px)`）或内阴影收敛，模拟**按压实体键**。
- **输入框 / 下拉**：深色底 + 内发光克制；focus 时**冷色描边 1～2px**。
- **进度与训练**：进度条使用冷色填充 + 深色槽；不确定进度用**细条纹动画**（低速），避免夸张弹跳。

---

### 5.5 动效与交互反馈

| 项目 | 要求 |
|------|------|
| 时长 | **120～200ms**，宜短不宜长 |
| 曲线 | `ease-out` 或 `linear`，**禁止**弹性回弹（spring/bounce） |
| 页面切换 | 淡入或极短位移（4～8px），避免大范围滑动 |
| 工具卡片展开 | 高度动画或淡入，与 3.4.2 时间线一致 |
| 告警 | 闪烁频率低，且提供**关闭/静音** |

**Hover / Active** 规则见 5.4；**禁用态**降低透明度 + 无 hover 放大，避免误触暗示。

---

### 5.6 关键页面与 Agent 界面强化

以下页面须在**同一套设计 token**（CSS 变量）下实现，保证全站「工控台」一致：

| 区域 | 设计要点 |
|------|-----------|
| **全局顶栏 / 侧栏** | 显示产品名、环境（可选）、当前用户；侧栏导航项**当前页**用冷色左边条 + 文字高亮；图标线性、硬朗。 |
| **项目列表 / 仪表盘** | 卡片式项目块；关键数字（子项目数、模型数）用大号数字 + 小标签；空状态用**线框图标 + 一句工程化文案**，避免插画。 |
| **子项目工作台** | Tab 为**分段控件**或下划线式，当前项冷色；训练区与数据区**分区线**明确；上传区为**虚线框 + 拖拽态冷色高亮**。 |
| **Agent** | **会话列表**为窄行列表 + 当前项冷色左边条；**新建/删除**为硬朗图标按钮。**上下文条**（项目/子项目）**铭牌式**或标签组，与 **3.4.3** 一致；**工具调用卡片**为深色面板 + 左侧状态色条（进行中青、成功绿、失败琥珀）；**终端**黑底绿字/青字（可切换主题），等宽字体；**附件列表**紧凑列表 + 小图标，且仅显示当前会话附件。 |
| **模态 / 危险操作** | 删除、训练中断等使用**琥珀标题条**或顶部警示带；主按钮在危险操作用**描边琥珀**而非纯红满屏。 |

---

### 5.7 技术实现建议（供开发落地）

- 使用 **CSS 变量**集中定义：`--bg-base`、`--surface`、`--accent`、`--accent-warn`、`--border-subtle`、`--text-primary` 等，便于暗色主题统一维护。
- **图表**（ECharts/Plotly）：深色主题坐标轴、网格线低对比；序列色以**冷色 + 琥珀**为主，避免默认高饱和彩虹。
- **响应式**：断点处侧栏可折叠为图标栏；Agent 三栏在窄屏下 **终端抽屉化**（见 **5.0** 中 Agent 布局建议）。
- **可访问性**：焦点可见（冷色环）；对比度满足 WCAG AA 在深色背景上的正文与控件；告警闪烁遵守 `prefers-reduced-motion` 时减弱或关闭动画。

## 6. 接口与数据模型（概要）

实现阶段可细化为 OpenAPI；此处仅列核心资源。

| 资源 | 主要操作 |
|------|-----------|
| `/projects` | CRUD |
| `/projects/{id}/knowledge/...` | 文档上传、删除、检索 |
| `/projects/{id}/subprojects` | 子项目 CRUD |
| `/projects/.../subprojects/{sid}/data` | 数据上传与版本 |
| `/projects/.../subprojects/{sid}/runs` | 启动训练、查询状态与结果 |
| `/projects/.../subprojects/{sid}/models` | 注册/列出/删除用户模型 |
| `/projects/.../subprojects/{sid}/models/{mid}/batch-predict` | 手动上传预测集文件，返回结果文件流（与旧版一致） |
| `/projects/.../subprojects/{sid}/artifacts` | 分析图与附件 |
| `/agent/sessions` | 创建、**列表、更新（重命名）、删除**会话（body/路径含 `project_id`、`sub_project_id` 等） |
| `/agent/sessions/{id}/attachments` | 上传预测集/数据文件至会话（供 Tool 引用） |
| `/agent/sessions`、`/agent/.../turn` | 流式对话与**可视化事件**（工具、文件、终端） |
| `/catalog/model-templates` | 预置模型目录 |

### 6.1 OpenAI 兼容 HTTP API（程序化调用）

平台对外提供与 **OpenAI Chat Completions** 兼容的 HTTP 接口，便于脚本、Notebook 与第三方工具通过 **同一套代理** 调用对话能力；`base_url` 指向部署主机下的 **`/v1/`** 前缀。

**约定**：

| 项 | 说明 |
|----|------|
| **Base URL** | `https://<hostname>/v1/`（末尾保留 `/v1/`，与官方 OpenAI Python SDK 习惯一致） |
| **API Key** | 用户在平台申请到的密钥，放入请求头 `Authorization: Bearer <api_key>`（SDK 自动处理） |
| **`model`** | 代理侧配置的**模型 ID**（字符串），与平台路由或后端实际模型一致；具体可选值以部署文档或 `/v1/models` 为准（若实现） |
| **与 Web 会话的关系** | 纯 `chat.completions` 调用默认**不绑定** Web UI 会话；若需与某 `project_id` / `sub_project_id` / `session_id` 对齐，由实现方通过**额外请求头**或 **metadata** 扩展（须在 OpenAPI 中单列，避免静默歧义） |

**Python 示例（官方 `openai` 库）**：

```python
import openai

client = openai.OpenAI(
    api_key="申请到的key",
    base_url="https://<hostname>/v1/",
)

response = client.chat.completions.create(
    model="模型ID",  # 发往代理的模型标识
    messages=[{"role": "user", "content": "Hello World"}],
)
print(response)
```

**实现提示**：服务端应对 `/v1/chat/completions` 实现与 OpenAI 接近的请求/响应 JSON 结构；流式响应若支持，使用 SSE `data: {...}` 分片。与平台 Agent（工具调用、RAG、会话附件）的深度集成可通过**同一代理**在网关层注入上下文，或在后续版本增加带 `session_id` 的扩展字段。

## 7. 验收标准（摘要）

- 可完成：创建项目 → 上传知识文档 → 建立子项目 → 选用预置模板训练 → 保存多个模型 → 上传/查看分析图。
- **手动预测**：对已保存模型上传预测集，成功下载带预测列的结果文件；列不匹配时错误提示明确。
- **Agent**：在选定 **项目 + 子项目** 后，界面展示上下文指示条；知识检索仅命中该项目知识库；工具列表仅含该子项目下模型（及系统工具）。
- **会话与上下文**：具备 **3.4.3** 所列会话列表、新建、切换、重命名、删除；项目/子项目选择器与上下文条常驻；切换会话后历史与附件不串会话。
- Agent 端到端：上传预测集附件 → 至少一次 **batch 推理 Tool** 成功 → 结果可下载或可在界面中预览摘要；**时间线**上可见工具调用卡片与终端输出。
- **OpenAI 兼容 API**：`POST /v1/chat/completions` 可用 **§6.1** 示例调用成功；密钥与 `model` 由部署方文档说明。
- 切换项目或子项目后，Agent 行为与工具列表随之变化，无跨项目数据泄露。
- 删除子项目后，对应模型 Tool 不可再调用。
- **UI**：整站采用 **§5** 工业控制台视觉规范（深色基底、冷色高光、琥珀警示、120～200ms 动效）；Agent 页具备上下文铭牌、工具卡片状态色条与终端区，无「纯聊天白盒」体验。
